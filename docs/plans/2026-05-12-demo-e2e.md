# Demo end-to-end: Go API → SQS → Python → Parquet

## Goal

Expand `examples/demo/` from a proto-rendering example into a complete polyglot exemplar that demonstrates the full value proposition of OpenEvents: a single YAML registry drives type-safe event production (Go) and type-safe event consumption (Python) across a real message bus, ending in queryable Parquet output.

## Non-goals

- No production hardening (retries, DLQs, autoscaling, observability).
- No alternative backends (only protobuf + SQS + Parquet).
- No CI integration of the e2e flow — Makefile is enough for local runs and an optional follow-up.
- No new events; the existing `checkout.started`, `checkout.completed`, `search.performed` cover all three.
- No change to the OpenEvents compiler, lock file, or codegen pipeline. This work consumes the existing pipeline, it doesn't modify it.

## Architecture overview

```
                                                                               |
```

## Wire format

| Element | Value |
|---|---|
| SQS message body | `base64(protobuf-binary)` |
| SQS message attribute `event_name` | e.g. `checkout.started@1` (self-describing; matches lock-file key format) |
| SQS message attribute `schema` | `openevents:com.acme.storefront/v1` |

Rationale: SQS bodies must be UTF-8; base64 keeps protobuf binary safe and stays inspectable from `aws sqs receive-message`. Routing via message attribute (not body framing) means the consumer can decide which generated Python class to instantiate **before** decoding bytes — defends against malformed payloads contaminating decoder state. The `@<version>` suffix on `event_name` matches the lock-file key format (`checkout.started@1`), so attribute values trace cleanly back to `openevents.lock.yaml`. The protobuf envelope still carries `event_version` as a typed int64 for consumers that need it post-decode.

## HTTP API (Go + Echo)

| Method | Path | Body | Response |
|---|---|---|---|
| `POST` | `/v1/events/checkout-started` | property JSON + `context` | `202 {"event_id","queue_url","message_id"}` |
| `POST` | `/v1/events/checkout-completed` | property JSON + `context` | `202` |
| `POST` | `/v1/events/search-performed` | property JSON + `context` | `202` |
| `GET`  | `/healthz` | — | `200 OK` |

**Body shape contract:** clients send `context` + event-specific properties only. The server fills `event_id` (`uuid.NewString()`), `event_ts` (`timestamppb.Now()`), `event_name`/`event_version` (from the route), and `client = {name: "demo-api", version: "0.1.0"}`. Clients never set envelope fields directly — this matches how a real production emitter would expose the API; the demo isn't a forwarder.

Per-handler flow:
1. `c.Bind(&req)` into a typed request struct (specific per event).
2. Hand-written `Validate()` per event in `eventmap/` returns a list of field-level errors. Required fields are derived from the registry (`required: true`) and hard-coded; tests assert that drift between registry and validator produces a failing case. Return 400 with `{errors: [{field, message}]}` if any.
3. Construct the protobuf message: fill `Context`, fill the event-specific `Properties`, set envelope (`event_name`, `event_version`, `event_id=uuid.NewString()`, `event_ts=timestamppb.Now()`, `client={name: "demo-api", version: "0.1.0"}`).
4. `proto.Marshal` → `base64.StdEncoding.EncodeToString`.
5. `publisher.Publish(ctx, eventName, body, attrs)`.
6. Return `202 {event_id, queue_url, message_id}`.

## Python consumer

Single-process polling loop, no async:

```python
while not stop_event.is_set():
    resp = sqs.receive_message(
        QueueUrl=queue_url,
        WaitTimeSeconds=20,
        MaxNumberOfMessages=10,
        MessageAttributeNames=["All"],
    )
    for msg in resp.get("Messages", []):
        name = msg["MessageAttributes"][ATTR_EVENT_NAME]["StringValue"]
        cls = EVENT_REGISTRY[name]
        decoded = cls()
        decoded.ParseFromString(base64.b64decode(msg["Body"]))
        row = MessageToDict(decoded, preserving_proto_field_name=True, including_default_value_fields=False)
        sink.append(name, row)
        sqs.delete_message(QueueUrl=queue_url, ReceiptHandle=msg["ReceiptHandle"])
    sink.maybe_flush()
```

`Sink.append(name, row)` appends to a per-event-name buffer. `Sink.maybe_flush()` writes parquet when `len(buffer) >= batch_size` (default 10) or `time.monotonic() - last_flush > flush_interval` (default 5s) — whichever first.

Parquet schema is **derived from the proto descriptors at module load** — not inferred per-batch. `consumer/schemas.py` walks `events_pb2.<Msg>.DESCRIPTOR.fields` once, maps proto types to polars dtypes, and builds a per-event `pl.Schema`:

```python
# consumer/schemas.py (sketch)
PROTO_TO_POLARS = {
    FieldDescriptor.TYPE_STRING: pl.Utf8,
    FieldDescriptor.TYPE_INT64:  pl.Int64,
    FieldDescriptor.TYPE_INT32:  pl.Int32,
    FieldDescriptor.TYPE_BOOL:   pl.Boolean,
    FieldDescriptor.TYPE_DOUBLE: pl.Float64,
    FieldDescriptor.TYPE_ENUM:   pl.Utf8,    # MessageToDict emits enum names
    FieldDescriptor.TYPE_MESSAGE: None,      # recurse for Timestamp / Context
}

def build_schema(msg_cls) -> pl.Schema:
    fields = {}
    for f in msg_cls.DESCRIPTOR.fields:
        fields[f.name] = _resolve_dtype(f)   # handles repeated → pl.List, message → pl.Struct
    return pl.Schema(fields)

EVENT_SCHEMAS: dict[str, pl.Schema] = {
    name: build_schema(cls) for name, cls in EVENT_REGISTRY.items()
}
```

When `Sink.flush(name, rows)` is called, it constructs `pl.DataFrame(rows, schema=EVENT_SCHEMAS[name])`. Missing optionals become `None` in the typed column instead of erasing the column's dtype. `context` is a `pl.Struct` derived from `Context.DESCRIPTOR`. This is the demo's "money shot" — the registry → proto → polars schema chain is the whole pitch.

The mapping table in `schemas.py` is the natural extension point. README points at it as the place to add custom types if the registry gains new ones.

Output: `_build/demo-output/<event_name>/<YYYYMMDDTHHMMSSZ>-<batch-id>.parquet`. (Slashes in event names like `checkout.started@1` are sanitized to underscores: `checkout_started_v1/...`.)

## Makefile orchestration

```makefile
PROTO_OUT   = ../../_build/demo-proto         # relative from services/api or services/consumer
PARQUET_OUT = ../../_build/demo-output
QUEUE_NAME  = storefront-events

.PHONY: gen up seed api consumer demo verify down clean

gen:
	cd ../.. && go run ./cmd/openevents generate proto ./examples/demo ./_build/demo-proto
	cd ../../_build/demo-proto && PATH="$$PWD/../../.tools/bin:$$PATH" ../../.tools/bin/buf generate .

up:
	docker compose up -d localstack
	# poll /_localstack/health until sqs == "available"

seed:
	docker compose exec -T localstack awslocal sqs create-queue --queue-name $(QUEUE_NAME) >/dev/null
	@echo "queue: $(QUEUE_NAME) ready"

api:
	cd services/api && go run .

consumer:
	cd services/consumer && uv run python -m consumer

demo:
	# scripted: gen, up, seed, run api in background, curl 3 samples, run consumer
	# 5 seconds, kill api, run verify
	@./scripts/demo.sh

verify:
	uv run --with polars python -c "import polars as pl; print(pl.read_parquet('_build/demo-output/**/*.parquet', glob=True))"

down:
	docker compose down -v

clean:
	rm -rf _build/demo-output _build/localstack
```

Note: `make demo` calls `scripts/demo.sh` (lives under `examples/demo/scripts/`) because composing background processes and curl in a Makefile gets unreadable fast.

### `scripts/demo.sh` shape

```bash
#!/usr/bin/env bash
set -euo pipefail

API_PID=""
CONSUMER_PID=""

cleanup() {
  [[ -n "$CONSUMER_PID" ]] && kill "$CONSUMER_PID" 2>/dev/null || true
  [[ -n "$API_PID"      ]] && kill "$API_PID"      2>/dev/null || true
  docker compose down -v >/dev/null 2>&1 || true
}
trap cleanup EXIT INT TERM

make gen
make up
# poll localstack health, max 60s
until curl -fsS http://localhost:4566/_localstack/health | grep -q '"sqs": "available"'; do
  sleep 1
done
make seed

(cd services/api && go run .) &
API_PID=$!
# wait for /healthz
until curl -fsS http://localhost:8080/healthz >/dev/null; do sleep 0.5; done

# Sample files are named by route: samples/checkout-started.json → /v1/events/checkout-started
for sample in samples/*.json; do
  path=$(basename "$sample" .json)
  curl -fsS -X POST "http://localhost:8080/v1/events/$path" \
       -H 'content-type: application/json' \
       --data-binary "@$sample"
  echo
done

(cd services/consumer && timeout 15s uv run python -m consumer --until-empty) &
CONSUMER_PID=$!
wait "$CONSUMER_PID" || true

make verify
```

`--until-empty` is a consumer flag: poll until two consecutive empty receives, then flush and exit. Gives `make demo` a clean termination instead of `timeout`-killing mid-flush.

## Python project setup (uv)

The generated Python tree (`_build/demo-proto/gen/python/`) is **not** importable as-is: `protoc_builtin: python` does not emit `__init__.py` files, and there is no `pyproject.toml`. We fix this in `make gen` so the consumer can depend on it as a normal editable package.

### `make gen` post-processing

After `buf generate`, the Makefile runs a small post-step that turns `gen/python/` into a proper Python package:

```bash
# 1. Make every directory a regular package.
find _build/demo-proto/gen/python -type d -exec touch {}/__init__.py \;

# 2. Drop a minimal pyproject.toml at gen/python root so uv can install it.
cat > _build/demo-proto/gen/python/pyproject.toml <<'EOF'
[project]
name = "openevents-demo-pb2"
version = "0.0.0"
requires-python = ">=3.11"
dependencies = ["protobuf>=5.0"]

[tool.setuptools.packages.find]
where = ["."]
include = ["com*"]

[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"
EOF
```

Both steps are scripted in the Makefile (or `scripts/postgen.sh` if the heredoc gets unwieldy), so contributors only ever run `make gen`.

## Tests

- **Go integration** (build tag `integration`): `publisher_test.go` boots a one-off ephemeral SQS client against LocalStack to assert a real round-trip. Skipped unless `go test -tags=integration ./...` is invoked, so the standalone `services/api` module stays hermetic.
- **Python unit**: `test_dispatch.py` feeds known pb-binary blobs into dispatch and asserts decoded dict shape; `test_sink.py` accumulates rows → writes to a `tmp_path` parquet → reads back → asserts columns; `test_attrs.py` is the contract test.
- **Cross-language contract**: `examples/demo/scripts/check-contracts.sh` greps both languages for the constants and fails if they drift. Called from a (future) CI job and listed in the README's "before committing" section.

## Graceful shutdown

The Python consumer installs a SIGINT/SIGTERM handler that sets `stop_event`. The poll loop checks it before each `receive_message` call and after each batch. On stop:
1. Flush any in-memory buffers to parquet (one final write per event type).
2. **Do not** delete any SQS messages that were received but not yet flushed — they return to the queue after the visibility timeout and are reprocessed by the next consumer.
3. Exit with status 0.

Expected shutdown latency: ≤10 seconds (one in-flight `receive_message` long-poll plus a final flush). The `--until-empty` flag used by `make demo` cooperates with this: it sets `stop_event` after two consecutive empty receives, so the consumer exits cleanly without needing an external signal.

## AWS SDK configuration

Both services use the standard `AWS_ENDPOINT_URL` env var pattern, which AWS SDK Go v2 (≥1.30) and boto3 (≥1.35) honor natively — no custom endpoint resolvers needed.

`.env.example` (committed):

```env

AWS_REGION=us-east-1
```

## Delivery semantics

The consumer is **at-least-once**, no deduplication. If a process crash sits between `parquet.write` and `sqs.delete_message`, the message is redelivered on the next poll and a duplicate row lands in parquet. The OpenEvents envelope already carries `event_id` (UUID set by the producer) for exactly this reason — `make verify` includes a `SELECT DISTINCT event_id` example so users see the pattern. A production consumer would add an `event_id`-keyed dedup store (Redis / DynamoDB); the demo deliberately omits it.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Go `replace` directive confuses new readers | README has a dedicated "Why the replace directive?" subsection with a 4-line explanation and a pointer to how a real project ships generated modules. |
| Polars 1.x schema inference changes on minor versions | Pin `polars>=1.0,<2` in pyproject.toml; tests assert column types explicitly. |
| LocalStack image pull on first `make up` is slow | README warns ("first `make up` pulls ~500MB"); `make demo` polls health for up to 60s. |
| Python `events_pb2` import path differs between IDE and `uv run` | `pyproject.toml` adds `tool.uv.sources.events-pb2 = path:...`; document that `make gen` must precede `uv sync`. |
| Drift between Go and Python constants | Contract tests in both; `scripts/check-contracts.sh` grep guard. |
| Generated proto path becomes stale if registry changes | `make gen` is the only entrypoint; README never tells users to invoke `buf generate` directly. |
