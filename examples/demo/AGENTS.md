# Agent Instructions — OpenEvents Demo

This directory is the **OpenEvents reference application** — a real working app demonstrating what a production OpenEvents-using service looks like, not an integration test fixture. The compiler under `internal/` is the tool being demonstrated; this directory consumes it.

The demo will eventually move to its own repository (working name: `acme-events`) once the compiler hits 1.0 / contract stability. Until then, treat it as if it were already standalone — the layout, imports, and build flow should reflect what a real consumer of OpenEvents would do, not what's convenient because we share a monorepo.

These instructions apply only to `examples/demo/**`. For compiler concerns see the root `CLAUDE.md`.

## Project Shape

- `registry/` — the YAML event registry (your source of truth). 12 events across `user` and `device` domains. The registry's `openevents.lock.yaml` pins protobuf wire numbers.
- `registry/.openevents/proto/` — generated proto sources produced by `openevents generate`. Not hand-edited.
- `services/api/` — Go HTTP service (Echo) that accepts JSON event payloads, validates them, builds protobuf envelopes, and publishes to LocalStack SQS. Own `go.mod`.
- `services/api/eventmap/<domain>/` — Per-domain bindings. `event_names.go`, `context.go`, and `<action>_request.go` are **openevents-generated**. `<action>.go` (ToProto), `helpers.go`, and `routes.go` are hand-written.
- `services/api/eventmap/pb/{user,device,common}/` — buf-generated protobuf Go types. (Migrating to `gen/go/` — see arc issue 07cn81.)
- `services/consumer/` — Python consumer that reads SQS, decodes protobuf envelopes, and writes Parquet. Own `pyproject.toml` (Python 3.11+).
- `samples/` — Sample JSON payloads matching each event's `*Request` JSON shape.
- `scripts/` — `demo.sh` runs the end-to-end flow; `postgen.sh` post-processes buf output; `verify.py` checks Parquet rows.
- `_build/` — LocalStack volume + binary build artifacts. Gitignored.

## Build and Run

```bash
# From this directory
make gen         # validate + lock check + openevents generate + buf generate
make up          # start LocalStack
make seed        # create the SQS queue
make api         # run the Go HTTP API
make consumer    # run the Python SQS consumer (separate terminal)
make demo        # one-shot: spin up, seed, send sample events, dump Parquet
make verify      # check Parquet output
make down        # stop LocalStack
make clean       # remove _build and gen outputs
```

The demo is **not** exercised by repo-root `go test ./...` — services have their own modules. Run their tests from inside each service directory:

```bash
cd services/api && go test ./...
cd services/consumer && uv run pytest
```

## Generation Pipeline

1. `openevents generate ./registry`
   - Writes proto sources to `registry/.openevents/proto/proto/com/acme/platform/...`
   - Writes Go bindings to `services/api/eventmap/{user,device}/` (event_names + context + per-event request structs + Validate methods)
   - Writes Python bindings to `services/consumer/src/consumer/` (event_names + context dataclasses)
2. `buf generate` (run inside `registry/.openevents/proto/`)
   - Writes Go protobuf types to `services/api/eventmap/pb/{user,device,common}/` (will move to `gen/go/` per D-7)
   - Writes Python protobuf bindings to `registry/.openevents/proto/gen/python/`
3. `postgen.sh` copies Go proto files into the API's eventmap tree and scaffolds the Python proto package.

## Hand-written vs Generated

| File | Status | Notes |
|------|--------|-------|
| `eventmap/<domain>/event_names.go` | Generated | One const per event. |
| `eventmap/<domain>/context.go` | Generated | `<Domain>Context` struct. |
| `eventmap/<domain>/<action>_request.go` | Generated | Request struct + Validate + enum value-sets. **Do not hand-edit.** |
| `eventmap/<domain>/<action>.go` | Hand-written | `ToProto()` + proto-enum maps. |
| `eventmap/<domain>/helpers.go` | Hand-written | `validateContext` + `contextToProto`. |
| `eventmap/<domain>/routes.go` | Hand-written | HTTP route wiring. |
| `eventmap/types.go` | Hand-written | Shared `FieldError`, `Route`, `BindBuild` generic. |
| `eventmap/pb/**/*.pb.go` | Buf-generated | Wire types. (Will move to `gen/go/` per D-7.) |

When a new event is added to the registry, only `eventmap/<domain>/<action>.go` (route + ToProto) is hand-written; the rest is generated by `make gen`.

## Implementation Guidelines

- Treat the demo as a real production app, not a test fixture. Code quality concerns are meaningful, not noise.
- Required primitive fields use pointer types (`*string`, `*int64`, `*bool`) so the validator can distinguish "field omitted from JSON" from "field set to its zero value". This is the CODEX-4 pattern, now applied uniformly by the openevents codegen.
- Generated files (`*_request.go`, `event_names.go`, `context.go`, `*.pb.go`) must not be hand-edited. If you need to change the generated shape, change the registry YAML or the openevents emitter under `internal/codegen/`.
- Run `make gen` after any change to the registry or to the openevents emitter before testing.
- Python consumer code is sync (boto3/polars/protobuf are sync). PEP 604 union types (`X | None`), not `Optional[X]`.
- Demo end-to-end run requires Docker (for LocalStack). Without Docker, run only the unit tests inside each service.

## Future: Standalone Repo Extraction

When the compiler hits 1.0 / contract stability, this directory extracts to its own repo (`acme-events`) via `git filter-repo`. Maintain the in-tree layout as if it were already standalone:

- Services depend on the demo's own go.mod paths (not the parent repo's).
- The demo's CLAUDE.md (this file) carries demo-specific agent guidance independent of the compiler.
- Generated artifacts (`gen/go/`, `gen/python/`) commit to the repo so `git clone && make` works without the buf+protoc toolchain on a fresh machine.

See arc issue `openevts-0yzp.07cn81` for the layout-migration plan.
